# 个性化旅游推荐系统开发规范

## 1. 代码规范

### 1.1 前端代码规范

#### 1.1.1 文件命名
- 组件文件：使用 PascalCase 命名，如 `MapComponent.vue`
- 工具文件：使用 camelCase 命名，如 `request.js`
- 页面文件：使用 PascalCase 命名，如 `Recommendation.vue`

#### 1.1.2 代码风格
- 使用 2 个空格缩进
- 变量和函数名使用 camelCase
- 组件名使用 PascalCase
- 常量使用全大写，单词间用下划线分隔
- 每行代码长度不超过 80 个字符
- 大括号使用 K&R 风格（左大括号与语句在同一行）

#### 1.1.3 Vue 组件规范
- 组件 props 必须使用类型验证
- 组件必须有明确的 name 属性
- 组件方法必须使用箭头函数或绑定 this
- 计算属性应使用 getter 函数
- 避免在模板中直接使用复杂表达式

### 1.2 后端代码规范

#### 1.2.1 文件命名
- 类文件：使用 PascalCase 命名，如 `UserService.java`
- 接口文件：使用 PascalCase 命名，如 `UserMapper.java`
- 配置文件：使用 snake_case 命名，如 `application.yml`

#### 1.2.2 代码风格
- 使用 4 个空格缩进
- 类名使用 PascalCase
- 方法和变量名使用 camelCase
- 常量使用全大写，单词间用下划线分隔
- 每行代码长度不超过 120 个字符
- 大括号使用 Allman 风格（左大括号独占一行）

#### 1.2.3 Java 代码规范
- 每个类都必须有类注释
- 每个方法都必须有方法注释
- 避免使用魔法数字，使用常量代替
- 异常处理必须规范，避免捕获异常后不处理
- 资源必须正确关闭，推荐使用 try-with-resources

## 2. 开发流程规范

### 2.1 版本控制
- 使用 Git 进行版本控制
- 主分支：main（生产环境）
- 开发分支：develop（开发环境）
- 特性分支：feature/xxx（功能开发）
- 修复分支：fix/xxx（bug 修复）

### 2.2 代码提交规范
- 提交信息必须使用中文
- 提交信息格式：`[类型] 描述`
  - 类型：feat（新功能）、fix（修复）、docs（文档）、style（格式）、refactor（重构）、test（测试）、chore（构建）
  - 描述：简洁明了，不超过 50 字符
- 每个提交应该只包含一个逻辑变更
- 提交前必须运行代码格式化工具

### 2.3 开发日志

#### 2.3.1 每日开发日志
- 每天结束工作前，必须在 `docs/Development Specification/DevelopmentLog.md` 中记录当日工作内容
- 日志格式：
  ```markdown
  ## YYYY-MM-DD
  - 完成的工作：
    - [x] 功能 1
    - [x] 功能 2
  - 遇到的问题：
    - 问题 1
  - 明天计划：
    - 计划 1
    - 计划 2
  ```

#### 2.3.2 功能开发日志
- 每个功能开发完成后，必须在 `docs/Development Specification/FeatureLog.md` 中记录功能开发详情
- 日志格式：
  ```markdown
  ## 功能名称
  - 功能描述：
  - 开发时间：YYYY-MM-DD 至 YYYY-MM-DD
  - 开发人员：
  - 实现细节：
  - 测试结果：
  - 备注：
  ```

## 3. 文档规范

### 3.1 技术文档
- 技术文档必须使用 Markdown 格式
- 文档结构必须清晰，使用适当的标题层级
- 代码示例必须使用代码块，标明语言
- 流程图必须使用 Mermaid 语法
- 表格必须使用 Markdown 表格语法

### 3.2 API 文档
- API 文档必须包含：接口路径、方法、功能描述、请求参数、响应格式
- API 响应格式必须统一，包含 code、data、message 字段
- API 文档必须及时更新，与实际代码保持一致

## 4. 测试规范

### 4.1 单元测试
- 前端：使用 Jest 进行组件测试
- 后端：使用 JUnit 进行服务层和控制器测试
- 测试覆盖率目标：核心功能 >= 80%

### 4.2 集成测试
- 测试 API 接口的正确性
- 测试前后端集成的完整性
- 测试系统各模块的协同工作

### 4.3 性能测试
- 测试线路规划算法的执行效率
- 测试系统在高并发下的响应时间
- 测试大数据量下的系统性能

## 5. 代码审查规范

### 5.1 代码审查流程
- 功能开发完成后，必须提交代码审查
- 代码审查至少需要 1 名其他开发人员参与
- 审查通过后才能合并到 develop 分支

### 5.2 代码审查要点
- 代码风格是否符合规范
- 代码逻辑是否正确
- 异常处理是否完善
- 性能是否优化
- 安全性是否考虑
- 测试是否覆盖

## 6. 环境配置规范

### 6.1 开发环境
- 前端：Node.js 16.x
- 后端：JDK 11
- 数据库：MySQL 8.0
- 缓存：Redis 7.0+（可选）

### 6.2 配置管理
- 敏感配置（如数据库密码）必须使用环境变量
- 不同环境（开发、测试、生产）的配置必须分开
- 配置文件必须添加到 .gitignore，避免提交敏感信息

## 7. 安全规范

### 7.1 代码安全
- 防止 SQL 注入攻击
- 防止 XSS 攻击
- 防止 CSRF 攻击
- 密码必须加密存储
- 敏感信息必须加密传输

### 7.2 数据安全
- 定期备份数据库
- 数据传输必须使用 HTTPS
- 敏感数据必须脱敏处理

## 8. 沟通规范

### 8.1 团队沟通
- 每日站会：15 分钟，汇报进度和问题
- 周会：每周一次，总结一周工作，规划下周任务
- 遇到问题及时沟通，避免影响开发进度

### 8.2 文档沟通
- 技术决策必须记录在文档中
- 重要变更必须通知所有团队成员
- 文档必须及时更新，反映最新状态

## 9. 版本发布规范

### 9.1 发布流程
- 功能开发完成后，合并到 develop 分支
- 测试通过后，合并到 main 分支
- 从 main 分支创建发布版本

### 9.2 版本号规范
- 版本号格式：x.y.z
  - x：主版本号，重大变更
  - y：次版本号，新增功能
  - z：补丁版本号，bug 修复

## 10. 其他规范

### 10.1 工具使用
- 代码编辑器：前端使用 VS Code，后端使用 IntelliJ IDEA
- 代码格式化：前端使用 Prettier，后端使用 IDE 内置格式化
- 包管理：前端使用 npm，后端使用 Maven

### 10.2 代码注释
- 复杂逻辑必须添加注释
- 公共接口必须添加文档注释
- 注释必须清晰明了，避免冗余

### 10.3 代码复用
- 重复代码必须提取为公共方法或组件
- 公共组件和工具类必须放在适当的目录
- 代码复用必须考虑可扩展性
